# 系统性文献综述技能

## 概述

此技能针对研究主题生成结构化的**系统性文献综述（SLR）**。给定一个主题查询，它搜索 arXiv，从每篇论文中并行提取结构化元数据（研究问题、方法论、主要发现、局限性），对全部论文进行主题综合，并以一致的引用格式生成最终报告。

**与 `academic-paper-review` 的区别：** 该技能对单篇论文进行深度同行评审。本技能对多篇论文进行广度优先的综合。如果用户提供一篇论文 URL 并要求"审阅这篇论文"，请路由到 `academic-paper-review`。

## 何时使用此技能

当用户需要以下任何内容时使用此技能：

- 关于某主题的文献调查（"调查 transformer 注意力变体"，"综述扩散模型的文献"）
- 跨多篇论文的综合（"最近关于 X 的论文说了什么"，"比较 Y 主题论文的方法论"）
- 具有一致引用格式的系统综述（"以 APA 格式对 Z 做 SLR"）
- 关于某主题的注释书目
- 在某时间窗口内的某领域研究趋势概述

**不要**在以下情况使用此技能：

- 用户提供恰好一篇论文并要求审阅（使用 `academic-paper-review`）
- 用户提出不需要综合多个来源的事实性问题（直接回答）
- 用户希望不需要学术严谨性的通用网络研究（使用标准网络搜索）

## 工作流程

工作流程包含五个阶段，按顺序执行。

### 第一阶段：规划

在进行任何检索之前，与用户确认以下内容。如果任何内容不清楚，提出**一个**涵盖缺失部分的澄清问题。不要一次问一个问题。

- **主题**：用通俗英语表达的研究领域（例如"transformer 注意力变体"）。
- **范围**：多少篇论文（默认 20，上限 50），可选时间窗口（例如"过去 2 年"），可选 arXiv 类别（例如 `cs.CL`、`cs.CV`）。
- **引用格式**：APA、IEEE 或 BibTeX（如果用户未指定且似乎不是为特定期刊写作，默认 APA）。
- **输出位置**：保存最终报告的位置（默认 `/mnt/user-data/outputs/`）。

如果用户说"50+ 篇"，礼貌地将其限制在 50 篇并解释综合质量超过这个数量会迅速下降——对于更大的调查，他们应该按子主题拆分。

### 第二阶段：搜索 arXiv

调用内置的搜索脚本。**不要**尝试用其他方式爬取 arXiv，**不要**编写自己的 HTTP 客户端——此脚本正确处理 URL 编码、Atom XML 解析和 id 规范化。

```bash
python /mnt/skills/public/systematic-literature-review/scripts/arxiv_search.py \
  "<主题>" \
  --max-results <N> \
  [--category <cat>] \
  [--sort-by relevance] \
  [--start-date YYYY-MM-DD] \
  [--end-date YYYY-MM-DD]
```

**重要——搜索前提取 2-3 个核心关键词。** 不要将用户的完整主题描述作为查询传入。在调用脚本之前，将主题精简为其 2-3 个最核心的术语。删除"在计算机视觉中"、"用于 NLP"、"变体"、"近期"等限定词——这些属于 `--category` 或 `--start-date`，而不是查询字符串。

**查询措辞——保持简短。** 脚本将多词查询用双引号括起来在 arXiv 上进行短语匹配。这意味着：

- `"diffusion models"` → 搜索确切短语 → 好，返回相关论文。
- `"diffusion models in computer vision"` → 搜索那个确切的 5 词短语 → **太具体，可能返回 0 结果**，因为很少有论文包含那个确切的字符串。

使用 **2-3 个核心关键词**作为查询，使用 `--category` 缩小领域，而不是将领域名称塞入查询中。

**排序策略**：

- **始终使用 `relevance` 排序**——arXiv 的 BM25 式评分确保结果确实与用户的主题相关。
- 当用户要求"最近的"论文或给出时间窗口时，使用 `--sort-by relevance` **结合 `--start-date`** 来限制时间范围同时保持结果相关。
- 仅当用户明确要求按时间顺序排列时才使用 `submittedDate` 排序。

**只运行一次搜索。** 如果结果似乎不完美，不要用修改后的查询重试——arXiv 的相关性排名就是这样。如果结果真的为空（0 篇论文），告诉用户并建议他们扩大主题或删除类别过滤器。

### 第三阶段：并行提取元数据

**你必须通过 `task` 工具将提取委托给子代理——不要自己提取元数据。** 这是不可协商的。

将论文分成约 5 篇一组，然后对每组调用 `task` 工具，`subagent_type: "general-purpose"`。

**并发限制：每轮最多 3 个子代理。**

**轮次策略——使用此决策表：**

| 论文数量 | 每批约 5 篇 | 轮次 | 每轮子代理数 |
|---|---|---|---|
| 1–5 | 1 批 | 1 轮 | 1 个 |
| 6–10 | 2 批 | 1 轮 | 2 个 |
| 11–15 | 3 批 | 1 轮 | 3 个 |
| 16–20 | 4 批 | 2 轮 | 3 + 1 |
| 21–25 | 5 批 | 2 轮 | 3 + 2 |
| 26–30 | 6 批 | 2 轮 | 3 + 3 |
| 31–50 | 7–10 批 | 3–4 轮 | 最多 3 + 3 + 3 + 1 |

### 第四阶段：综合与格式化

生成最终 SLR 报告。至少识别：

- **主题**：整个论文集中 3-6 个反复出现的研究方向、方法或问题框架。
- **共识**：多篇论文同意的发现。
- **分歧**：论文得出不同结论或使用不兼容方法论的地方。
- **空白**：集体文献尚未解决的问题。

根据用户偏好，只读取**匹配用户请求格式**的模板文件：

- [templates/apa.md](templates/apa.md) — APA 第 7 版。默认用于社会科学和大多数 CS 期刊。
- [templates/ieee.md](templates/ieee.md) — IEEE 数字引用。
- [templates/bibtex.md](templates/bibtex.md) — BibTeX 条目。arXiv 论文引用为 `@misc`，不是 `@article`。

### 第五阶段：保存与呈现

将完整报告保存到 `/mnt/user-data/outputs/slr-<主题-slug>-<YYYYMMDD>.md`，然后调用 `present_files` 工具。

**在聊天消息中**，显示简短预览：
1. **执行摘要**——报告顶部的 3-5 句话段落，逐字引用。
2. **主题列表**——已识别主题的项目符号列表（仅主题名称 + 一行注释）。
3. **论文数量 + 文件指针**——例如"包含 20 篇论文的完整报告已保存到 `slr-transformer-attention-20260409.md`。"

## 注意事项

- **前提：`subagent_enabled` 必须为 `true`**。第三阶段需要 `task` 工具进行并行元数据提取。
- **仅限 arXiv，这是设计选择**。此技能不查询 Semantic Scholar、PubMed 或 Google Scholar。
- **50 篇论文的硬上限**。超过 50 篇的调查综合质量会下降，最好按子主题拆分。
- **子代理结果是字符串，不是对象**。在解析 JSON 有效载荷之前，始终去除 `Task Succeeded. Result: ` / `Task failed.` / `Task timed out.` 前缀。
- **综合，不是列表**。最终报告必须识别主题并跨论文比较发现。
