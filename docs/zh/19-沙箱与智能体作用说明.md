# 沙箱与智能体作用说明

本文用于回答两个核心问题：
1. 沙箱在 DeerFlow 中什么时候会被使用？
2. 创建智能体（含子代理）的主要作用是什么？

---

## 1. 沙箱什么时候使用

在一次会话（thread）中，只要 Agent 需要执行工具能力，沙箱就会被使用，典型场景包括：

- 执行命令：`bash`
- 读写文件：`read_file` / `write_file` / `str_replace`
- 浏览目录与检索：`ls` / `glob` / `grep`

执行流程可概括为：

1. 用户发送消息。
2. `SandboxMiddleware` 在运行前为当前 thread 获取（或复用）沙箱。
3. Agent 调用工具时，命令和文件操作都在沙箱视图内执行。
4. 本轮结束后，系统按 provider 生命周期管理沙箱资源。

> 结论：沙箱不是“可选附加功能”，而是 Agent 执行命令与文件操作的运行基础设施。

---

## 2. 沙箱的主要功能

### 2.1 统一执行环境

对 Agent 暴露统一虚拟路径，屏蔽宿主机真实路径差异：

- `/mnt/user-data/workspace`：工作区（读写）
- `/mnt/user-data/uploads`：上传文件目录
- `/mnt/user-data/outputs`：输出目录
- `/mnt/skills`：技能目录（通常只读）

### 2.2 会话隔离与安全边界

- 以 thread 为粒度隔离工作数据，避免不同会话相互污染。
- 通过路径映射和校验减少越界访问风险。
- 在 Docker/AIO 等 provider 下可获得更强的进程级隔离。

### 2.3 工具能力承载

沙箱承接 Agent 的实际“动手能力”：执行 shell、读写代码、检索文件、产出结果文件等。

---

## 3. 创建智能体的作用

在 DeerFlow 中，“创建智能体”本质是创建一套可复用的任务执行配置（角色、工具、模型、技能、限制）。

主要价值如下：

1. **职责专业化**
   - 为不同场景定义不同角色（如代码助手、分析助手、运营助手）。

2. **能力可控**
   - 可限制工具集合、模型、提示词，降低误用风险。

3. **效率提升**
   - 将复杂任务拆分给子代理并行处理，主代理负责协调与汇总。

4. **结果稳定**
   - 固定系统提示词和技能后，输出风格与执行路径更一致。

---

## 4. 主代理与子代理的协作关系

- **Lead Agent（主代理）**：理解用户目标、规划步骤、决定是否委派。
- **Subagent（子代理）**：执行被委派的具体任务（如命令执行、专项分析）。
- **Sandbox（沙箱）**：为主代理与子代理提供共享但受控的文件与命令执行环境。

可理解为：

- 主代理负责“想清楚做什么”；
- 子代理负责“并行把事情做完”；
- 沙箱负责“安全地把动作落地”。

---

## 5. 一句话总结

**沙箱是 DeerFlow 中 Agent 进行命令执行与文件操作的运行底座；创建智能体是把角色、能力和约束产品化，用于更稳定、更高效地完成复杂任务。**
