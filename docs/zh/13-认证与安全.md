## 1. 认证系统概述

DeerFlow 内置完整的多用户认证模块，将其从「本地单用户工具」提升为「可多用户部署的 Agent 运行时」。

**设计目标**：
- 默认强制认证（除健康检查、文档和 auth bootstrap 端点外）
- 服务端持有所有权：客户端 metadata 不能声明 `user_id` 或 `owner_id`
- 隔离默认开启：数据文件、记忆、自定义 Agent 配置按当前用户解析
- 旧数据可升级：无认证版本遗留的 thread 可以在 admin 存在后迁移到 admin

**当前限制**：
- OAuth 端点仅为占位符，尚未实现第三方登录
- 角色仅有 `admin` 和 `user`，尚未实现细粒度 RBAC
- 登录限速是进程内字典，多 worker 下不是全局精确限速

## 2. 认证流程

```
浏览器请求
    │
    ▼
CSRFMiddleware
  ├── 验证 Origin 头（仅限允许的来源）
  └── 验证 csrf_token cookie 与 X-CSRF-Token 请求头（double-submit 模式）
    │
    ▼
AuthMiddleware
  ├── 从 access_token cookie 中读取 JWT
  ├── 验证签名、过期时间和 token_version
  └── 写入 request.state.user + request.state.auth
      ↓
  deerflow.runtime.user_context ContextVar（请求作用域）
      ↓
  Repositories / Files / Memory（自动按当前用户解析）
```

## 3. 用户数据模型

**表**：`users`（通过 SQLAlchemy 持久化）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | str | 用户主键，JWT `sub` 使用该值 |
| `email` | str | 唯一登录名 |
| `password_hash` | str | bcrypt hash，OAuth 用户可为空 |
| `system_role` | str | `admin` 或 `user` |
| `needs_setup` | bool | reset 后要求用户完成邮箱/密码设置 |
| `token_version` | int | 改密码或 reset 时递增，废弃旧 JWT |

## 4. Auth 路由

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/initialize` | 首次初始化 admin（仅当无用户时可用）|
| POST | `/api/auth/login` | 登录，返回 access_token cookie |
| POST | `/api/auth/logout` | 登出，清除 cookie |
| GET | `/api/auth/me` | 获取当前用户信息 |
| POST | `/api/auth/register` | 注册新用户（需 admin 授权）|
| POST | `/api/auth/change-password` | 修改密码 |
| POST | `/api/auth/reset-admin` | 重置 admin 密码（写 0600 凭据文件）|

## 5. 用户隔离范围

认证成功后，所有下层操作自动按当前用户隔离：

```python
# ContextVar 是核心边界
# Gateway 写入身份，下层只读取，不反向依赖 app.gateway.auth 具体类型

# 文件系统隔离
backend/.deer-flow/users/{user_id}/threads/{thread_id}/user-data/
backend/.deer-flow/users/{user_id}/memory.json
backend/.deer-flow/users/{user_id}/agents/

# 数据库隔离（仓储层自动注入 user_id）
threads → 按 owner_id 过滤
feedback → 按 user_id 过滤
```

## 6. 内部服务认证（IM 渠道）

IM 渠道（Feishu、Slack、Telegram、钉钉）通过 Gateway 内部 SDK 客户端发起请求时，注入进程本地的内部认证令牌，附带匹配的 CSRF cookie/header 对，无需浏览器 session cookie。

## 7. 护栏（Guardrails）

护栏系统在工具调用前提供策略驱动的授权检查，是对沙箱隔离的语义层补充。

```
沙箱：进程隔离（但沙箱内的 bash 仍可执行任意命令）
人工确认：需要 ask_clarification，不适合自动化工作流
护栏：确定性策略授权，无需人工干预
```

### 护栏位置（中间件链第 6 位）

```
5. GuardrailMiddleware ← 评估每次工具调用
     │
     ▼
┌──────────────────────┐
│    GuardrailProvider │ ← 可插拔：实现 evaluate / aevaluate 协议
└──────────────────────┘
     │
┌────┴────┐
│         │
ALLOW    DENY
│         │
工具正常  Agent 收到：
执行      "Guardrail denied: rm -rf blocked"
```

### 内置提供者：AllowlistProvider

```yaml
guardrails:
  enabled: true
  provider:
    use: deerflow.guardrails.allowlist:AllowlistProvider
    config:
      allowed_tools:
        - bash
        - read_file
        - write_file
        - ls
      # 可选：允许特定参数模式
      rules:
        bash:
          deny_patterns:
            - "rm -rf"
            - "sudo"
            - "curl.*|.*bash"
```

### 实现自定义护栏提供者

```python
# my_guardrail.py
from deerflow.guardrails.base import GuardrailProvider, GuardrailDecision

class MyGuardrailProvider(GuardrailProvider):
    async def aevaluate(
        self,
        tool_name: str,
        tool_args: dict
    ) -> GuardrailDecision:
        # 自定义授权逻辑
        if tool_name == "bash" and "rm -rf" in tool_args.get("command", ""):
            return GuardrailDecision(
                allow=False,
                reason="危险命令被拦截：rm -rf"
            )
        return GuardrailDecision(allow=True)
```

```yaml
guardrails:
  enabled: true
  provider:
    use: my_guardrail:MyGuardrailProvider
```

## 8. 安全建议（生产环境）

来源：[`README.md`](../../README.md) 的安全使用章节

| 风险 | 建议 |
|------|------|
| 本地沙箱无进程隔离 | 生产环境使用 Docker 沙箱（`sandbox.use: aio`）|
| Agent 可执行任意 bash | 开启护栏，限制危险命令 |
| 默认 CORS 策略 | 通过 `GATEWAY_CORS_ORIGINS` 明确允许的来源 |
| API 文档暴露 | 生产环境设置 `GATEWAY_ENABLE_DOCS=false` |
| 公网暴露 | 务必开启认证，不要在无认证情况下公网暴露 |
| Artifact XSS | 活跃内容类型（HTML/SVG）强制为下载附件，已内置保护 |

## 9. 认证模式切换

**无认证模式（开发/测试）**：

默认情况下，未配置认证时，`get_effective_user_id()` 回退到 `"default"`，所有数据存储在 `users/default/` 下。

**开启认证（生产）**：

配置 `database`、`checkpointer` 等，并通过 `/api/auth/initialize` 创建第一个 admin 用户。
